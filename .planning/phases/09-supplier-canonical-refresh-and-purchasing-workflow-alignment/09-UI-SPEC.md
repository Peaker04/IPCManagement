# Phase 09 — UI Design Contract

> Visual and interaction contract for supplier canonical refresh and the end-to-end purchasing workflow. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope and Design Intent

This phase is a targeted operational alignment, not a visual redesign. Preserve the current `/weekly-menu`, `/approvals`, `/purchasing` and Warehouse routes, authorization, IPC light theme, compact controls, small radii, Vietnamese labels and server-backed RTK Query state.

The purchasing surface must answer, in this order: which week is selected, which service date needs action, whether material demand is approved, what shortage remains, what supplier evidence exists, which price exception blocks progress, whether the purchase request is approved, whether supplier-split orders exist and how much Warehouse has received.

Visual dials are fixed at variance 3/10, motion 2/10 and density 8/10. Do not introduce a marketing layout, hero treatment, decorative gradient, glass effect, oversized typography, new font, second UI kit or client-only workflow state.

### Explicit UI boundary

- The audited 20.7 workbook refresh is a Development/sample reconciliation path. Phase 09 does not add a production workbook upload, local-path input or Admin import page.
- Operational selectors show only canonical, active suppliers returned by the server. SUMMARY headers, pseudo-suppliers and unreferenced placeholders never appear as choices.
- Historical documents may show an inactive or superseded supplier snapshot, but it is read-only and labelled `Nhà cung cấp lịch sử` or `Ngừng hoạt động`; it is never silently remapped in the UI.
- Unknown or ambiguous ingredient, supplier, unit or date data blocks the relevant action with visible evidence. The UI never guesses, clamps a date or defaults a unit to `KG`.

### Existing primitives to reuse

- Page shell: `OperationalFrame`, `CommandBar`, `ContextStrip`.
- Guided navigation and panels: `ViewSwitcher` only where it can expose real tab semantics, `SectionPanel`, `InlineAlert`, `StatusBadge`.
- Data scale: `TableViewport`, `PaginationBar` or `CursorPaginationBar`; totals come from the server and states stay inside the same viewport.
- Confirmation and forms: existing `Dialog`, `DialogContent`, `DialogTitle`, `DialogDescription`, `Button`, `Select`, `Input`, `Textarea` and `ConfirmDialog` patterns.
- Phase-local composition may introduce `PurchasingWorkflowGuide`, `ServiceDateGroup`, `SupplierEvidencePanel`, `PriceExceptionPanel` and `ReceivingProgress`. Do not modify shared table or pagination primitives unless a separately reviewed impact analysis proves it necessary.

## Information Architecture

### Preserved route ownership

| Route | Phase 09 responsibility |
|-------|-------------------------|
| `/weekly-menu` | Show material-demand approval status for the selected service date and link to the matching approval work |
| `/approvals` | Make material demand and price exception first-class, resolvable approval targets with history |
| `/purchasing` | Own week/date scope, supplier and price decisions, purchase-request submission, purchase-order creation and read-only receiving progress |
| Warehouse route | Own actual receipt entry: destination warehouse, quantity, lot, manufacture date and expiry date |

Do not move receipt mutation fields back into Purchasing. Do not make Weekly Menu or Purchasing perform a manager approval on behalf of `/approvals`.

### Purchasing page anatomy

The page uses one full-width vertical workbench in this fixed order:

1. `CommandBar`: week-start control, previous/next week, `Tuần hiện tại` and the primary eligible action for the selected service date.
2. `ContextStrip`: `Ngày cần xử lý`, `Nhu cầu chờ duyệt`, `Ngoại lệ giá`, `Đơn chờ nhập`.
3. `PurchasingWorkflowGuide`: six compact stages with visible status and one current stage.
4. `ServiceDateGroup`: server-paged dates nested under the selected week, sorted ascending.
5. One expanded service-date detail panel; all mutation controls and tables belong to this panel.
6. Contextual feedback region and pagination; no detached document rail or five independent top-level tabs.

The route remains `/purchasing`. Week, selected service date and selected stage may be represented in query parameters so refresh/back/forward restores context. RTK Query remains the authority for server state; local state owns only the selected week/date/stage and open dialog/form values.

### Week and service-date hierarchy

- The primary filter is a Monday-based week range displayed as `Tuần 20/07/2026 - 26/07/2026` in `vi-VN` format.
- Each date row shows weekday/date, total shortage lines, total shortage quantity by unit without invalid cross-unit aggregation, demand approval status, supplier-ready count, blocking exception count, purchase-request status, order count and receiving progress.
- A purchase request is always tied to one service date and visible scope `Cả ngày (FULLDAY)`. Do not expose a shift selector for this action.
- Only one service date is the active editing context. Changing dates preserves unsaved-form protection and never submits the first row implicitly.
- Empty dates are omitted only when the server confirms there is no demand, shortage or workflow document. A zero-shortage approved date may remain visible as `Không cần mua` for traceability.
- Tables stay server-paged at eight rows per page unless the existing endpoint contract supplies a different bounded page size. Filtering display rows never changes server totals or mutation payloads.

## Guided Workflow Contract

The six stages are navigable status controls, not five disconnected content tabs. Completed stages remain inspectable; a later stage is disabled until its server prerequisites pass. Disabled stages show a visible reason, not a tooltip-only explanation.

| Stage | Label | Entry condition | Complete when | Primary action |
|------:|-------|-----------------|---------------|----------------|
| 1 | `Nhu cầu đã duyệt` | Finalized servings and date-specific demand exist | Manager approval status is `Đã duyệt` | `Mở phê duyệt nhu cầu` when pending |
| 2 | `Chọn nhà cung cấp và giá` | Demand is approved and shortage is actionable | Every shortage line has an explicitly confirmed supplier, proposed price and delivery date | `Xác nhận nhà cung cấp` |
| 3 | `Xử lý ngoại lệ giá` | Supplier and proposed price exist | Every variance above 15% is approved; rows at or below 15% need no exception | `Gửi duyệt ngoại lệ giá` |
| 4 | `Gửi đề xuất mua` | Stages 1–3 pass and one draft is explicitly selected | The purchase request has been submitted | `Gửi đề xuất mua` |
| 5 | `Duyệt và tạo đơn` | Purchase request is submitted | Manager approved it and supplier-split purchase orders were created | `Tạo đơn đặt hàng` |
| 6 | `Theo dõi nhập kho` | At least one order exists | Warehouse received all ordered quantities | `Mở màn hình nhập kho` |

Status vocabulary is fixed:

- Demand: `Chưa tạo`, `Chờ duyệt`, `Đã duyệt`, `Từ chối`.
- Supplier: `Chưa chọn`, `Đã xác nhận`.
- Price: `Đủ căn cứ`, `Cần duyệt ngoại lệ`, `Chờ duyệt`, `Đã duyệt`, `Từ chối`.
- Purchase request: `Bản nháp`, `Chờ duyệt`, `Đã duyệt`, `Từ chối`.
- Purchase order: `Chưa tạo`, `Đã tạo`.
- Receiving: `Chưa nhận`, `Nhận một phần`, `Đã nhận đủ`.

## Cross-Route Interaction Contracts

### Weekly Menu demand status

- Place one compact status row beside the existing selected date/demand controls: label `Phê duyệt nhu cầu`, a `StatusBadge`, document code when available and one contextual link.
- `Chờ duyệt` links to `Mở hàng đợi duyệt`; `Đã duyệt` links to `Mở thu mua`; `Từ chối` shows the manager reason and keeps the demand regeneration action with its existing owner.
- The link carries the exact material-demand target and service date so `/approvals` opens the matching record; never link to an unfiltered queue when an ID exists.
- Do not show purchase actions before demand approval.

### Approval queue records

Material demand and price exception are first-class target types. Reuse `ApprovalQueue`, but each record must expose its target type in visible Vietnamese.

Material-demand record fields:

- `Nhu cầu nguyên liệu`, document code, service date, `Cả ngày (FULLDAY)`, source production/serving document, shortage-line count and top bounded material summary.
- Actions: `Duyệt nhu cầu` and `Từ chối nhu cầu`.

Price-exception record fields:

- `Ngoại lệ giá`, ingredient, supplier, reference price, proposed price, signed variance percentage, evidence type/date, purchaser reason and linked proposal/order code.
- Actions: `Duyệt ngoại lệ` and `Từ chối ngoại lệ`.
- A `price-alert` without `targetType`, `targetId` and working decision actions is forbidden.

Approval history shows decision, actor, timestamp and reason. After a decision, focus returns to the originating record if it remains on the page; otherwise focus returns to the queue heading and the result is announced.

### Supplier selection and evidence

- A suggestion is evidence, not a selection. No supplier is preselected merely because it is first or active.
- Each shortage line displays `Chưa chọn nhà cung cấp` until the user opens the selection dialog and confirms one canonical supplier.
- Supplier options show canonical name and one evidence line: `Báo giá hiệu lực đến {date}` or `Phiếu nhập gần nhất ngày {date}`. If neither exists, the supplier is not suggested as evidence-backed.
- The dialog visibly repeats ingredient, quantity/unit, service date, supplier, reference price, proposed price, evidence and delivery date.
- Primary CTA: `Xác nhận nhà cung cấp`. Success: `Đã xác nhận nhà cung cấp cho {ingredient}.`
- Package unit `BỊCH` displays package size when conversion is relevant, for example `1 BỊCH = 10 CÁI`. A plain `BỊCH` remains valid only when no cross-unit conversion is needed.
- Missing or ambiguous unit evidence shows `Đơn vị chưa được chuẩn hóa. Không thể tiếp tục mua dòng này.` and blocks confirmation.

### Price exception

- Variance is calculated by the server. The UI displays the reference and proposed values and does not independently decide whether an exception exists.
- Strict threshold: a variance greater than 15% requires approval; exactly 15% does not.
- The blocking panel displays reference price, proposed price, variance, evidence, bounded required reason and current manager decision.
- CTA is `Gửi duyệt ngoại lệ giá`. While pending, show `Đang chờ duyệt ngoại lệ giá` and disable purchase-request submission for that line/date.
- A rejected exception preserves audit history and offers two recovery paths: `Thay đổi nhà cung cấp hoặc giá` and `Gửi lại ngoại lệ` after evidence/reason changes.

### Purchase request and purchase order

- `Tạo đề xuất mua` is available only for an approved material demand with actionable shortage in the selected service date.
- Creation dialog lists approved candidates only and requires explicit selection. Do not default to the first paged candidate.
- `Gửi đề xuất mua` requires an explicitly selected draft. The selected code, service date, line count and total estimated amount are repeated beside the CTA.
- Manager approval remains on `/approvals`; Purchasing shows read-only status and a direct `Mở phê duyệt đề xuất` link.
- `Tạo đơn đặt hàng` becomes available only for an approved purchase request with confirmed suppliers and approved price exceptions. Confirmation previews the supplier split and order count.
- Success copy: `Đã tạo {N} đơn đặt hàng theo nhà cung cấp.`

### Warehouse-owned receiving

- Purchasing displays each order's ordered, received and remaining quantities and status. No quantity, warehouse, lot or date input is editable on this route.
- `Mở màn hình nhập kho` links to the exact order on Warehouse.
- Warehouse receipt form fields are: `Kho nhận`, `Số lượng nhận`, `Số lô`, `Ngày sản xuất`, `Hạn sử dụng`. Quantity and warehouse are required; lot/manufacture/expiry follow server validation and ingredient policy.
- Partial receipt is supported. The CTA is `Ghi nhận nhập kho`; success updates Purchasing read-only progress after query invalidation.
- Receipt quantity greater than remaining quantity, expiry before manufacture date or missing required lot/date data renders an adjacent field error and blocks save.

## Stable Table and State Geometry

- All purchasing line tables use `TableViewport`, native table semantics, an opaque sticky header and local horizontal scrolling. The page body never scrolls horizontally.
- Desktop table viewport height is 480px for line-heavy stages; narrow viewport height is 400px. Loading, error, empty and no-result states keep the same outer height and column header.
- Row actions use `inline-flex`, `flex-flow: row nowrap`, 4px gap and `white-space: nowrap`. On narrow screens, the table scrolls locally instead of wrapping action labels.
- Service-date rows and line tables use stable server IDs as keys. After a mutation, focus returns to the same logical date/line action when it remains available.
- Loading renders real headers plus fixed skeleton rows. Empty/error states render inside the table body or the same bounded panel, never as a differently sized replacement card.
- Pagination is visible whenever server metadata indicates more rows. Page changes do not reset the selected week or service date unless that date is no longer in scope.

## State and Feedback Contract

| State | Exact visible contract | Interaction contract |
|-------|------------------------|----------------------|
| Week loading | `Đang tải quy trình thu mua trong tuần...` | Keep week controls visible; disable duplicate navigation |
| No approved demand | `Chưa có nhu cầu đã duyệt trong tuần này.` plus `Hoàn tất nhu cầu nguyên liệu và mở Phê duyệt để tiếp tục.` | Show `Mở phê duyệt` when a pending record exists |
| No shortage | `Không có thiếu hụt cần mua cho ngày này.` | Stage guide marks the date complete with `Không cần mua` |
| Supplier evidence missing | `Chưa có báo giá hiệu lực hoặc phiếu nhập hợp lệ cho nguyên liệu này.` | Do not auto-select; preserve line as blocked |
| Price exception pending | `Ngoại lệ giá đang chờ quản lí duyệt.` | Disable submit for the affected date and link to approval |
| Purchase request rejected | `Đề xuất mua đã bị từ chối.` plus the manager reason | Preserve draft/history and expose correction path |
| Receiving partial | `Warehouse đã nhận {received}/{ordered} {unit}. Còn {remaining} {unit}.` | Purchasing stays read-only; Warehouse link remains available |
| Load error | `Không tải được quy trình thu mua. Kiểm tra kết nối và thử lại. Các lựa chọn chưa được lưu.` | Contextual `Thử lại`; preserve week/date selection |
| Mutation error | `Chưa thể lưu thay đổi.` plus server-safe reason | Keep dialog/form open and preserve entered values |
| Success | Specific completed action plus document/supplier code | Refresh affected RTK Query tags and return focus |

Use `role="alert"` only for blocking errors or failed mutations. Nonurgent loading, completion and progress updates use `role="status"` or `aria-live="polite"`. Never communicate state through color alone.

## Destructive and Confirmation Contract

- `Từ chối nhu cầu` and `Từ chối ngoại lệ` use the existing accessible dialog/confirmation pattern. The reason is trimmed, nonblank, adjacent to its error and required before enabling the destructive CTA.
- Safe action labels are contextual: demand rejection uses `Giữ nhu cầu`, price-exception rejection uses `Giữ ngoại lệ giá`, supplier confirmation uses `Quay lại chọn nhà cung cấp`, purchase-request submission uses `Giữ bản nháp` and purchase-order creation uses `Quay lại kiểm tra đơn`.
- Initial focus is the contextual safe action named above. The reject CTA uses the destructive token; approval and supplier confirmation do not.
- While a noncancellable mutation is pending, Escape, backdrop dismissal and close controls are disabled. Before mutation starts, Escape performs the same safe return as the visible contextual safe action.
- Activating the contextual safe action or pressing Escape returns focus to the exact trigger. On success, return to the updated record or stable queue/workbench heading and announce the result.
- Supplier confirmation, purchase-request submission and purchase-order creation use confirmation because they have operational consequences, but they use primary/warning styling rather than destructive red.
- No canonical workbook apply or supplier hard-delete confirmation is exposed in the production UI in this phase.

## Accessibility and Keyboard Contract

- Guided stages use an ordered list. Interactive stage controls expose current, complete, blocked and disabled states in text; if implemented as tabs they must use correct `tablist`, `tab`, `tabpanel`, `aria-selected` and linkage semantics.
- Service-date expanders expose `aria-expanded` and `aria-controls`. Keyboard order follows week controls, context strip links, stage guide, service dates, line table and current-stage actions.
- Dialogs have programmatically associated title and description. Field errors use `aria-describedby`; the first invalid field receives focus after submit.
- Table viewports are focusable named regions. Keep native table semantics; do not add a custom spreadsheet keyboard model.
- Icon-plus-text controls hide the icon from assistive technology. Icon-only controls require an accessible name and at least a 36px desktop/44px mobile hit area.
- Disabled stages and CTAs have a visible prerequisite explanation available without hover.
- Preserve 4.5:1 normal-text contrast and 3:1 large-text, border and state-indicator contrast.
- Reduced motion removes nonessential transitions. No animation is required to understand completion or failure.

## Responsive Contract

- At 1280px and wider, week controls and current action share the `CommandBar`; the six-stage guide may remain one row if every label fits without truncation.
- Below 1280px, the guide wraps into a two- or three-column ordered grid. Stage order remains left-to-right, top-to-bottom.
- Below 768px, week controls, actions, date summaries and stage controls stack full width. Control targets are at least 44px high.
- Service-date summaries become stacked key/value rows on mobile; detailed line tables remain horizontally scrollable inside `TableViewport`.
- Test at 1365x900, 1280x900, 768x1024 and 390x844. Document width must not exceed viewport width by more than two pixels.

## Verification Contract

### Component and integration checks

- Only approved material demand can be selected to create a purchase request.
- Week/date/stage URL state restores correctly after refresh and never selects the first paged draft or supplier implicitly.
- Supplier suggestions come only from an effective quotation or latest valid matching receipt; absence leaves the line unselected.
- Variance greater than 15% creates a resolvable approval target; exactly 15% does not.
- Pending/rejected price exceptions block submission and expose the specified recovery actions.
- Purchase-order creation previews and preserves supplier split.
- Purchasing has no receipt mutation fields; Warehouse captures warehouse, quantity, lot, manufacture and expiry values.
- Partial/full receiving progress invalidates and refreshes Purchasing read-only state.
- Material-demand and price-exception approval records have working approve/reject actions and append visible history.

### Browser and geometry checks

- At all target viewports, page-level horizontal overflow is absent and tables own horizontal scrolling.
- Stage labels do not overlap, truncate essential status text or reorder under wrapping.
- Loading, empty, error and populated fixtures keep the same table/panel geometry within one pixel at the same viewport.
- Keyboard navigation reaches week controls, every enabled stage, service-date expander, tables, actions and dialogs with visible focus.
- Every dialog's contextual safe action and submit action return focus correctly; pending noncancellable mutations cannot be dismissed accidentally.
- Status remains understandable in grayscale and with color removed.

### Existing quality gates

- `npm run lint --workspace frontend`
- `npm run build --workspace frontend`
- `npm run test:unit --workspace frontend`
- `npm run test:smoke --workspace frontend`
- `npm run test:ui-audit --workspace frontend`
- `npm run test:visual --workspace frontend`

Extend the existing route smoke, UI audit and visual route suites. Do not create a second browser harness.

## Forbidden Patterns

- Five disconnected purchasing tabs without a visible ordered workflow and prerequisite status.
- First-active-supplier fallback, preselected supplier suggestion or first-paged-draft submission.
- Purchase actions for unapproved material demand.
- A price warning with no `targetType`, `targetId`, approve/reject actions and history.
- Client-computed approval eligibility, variance threshold or authoritative workflow totals.
- Receipt form fields or receipt mutations on Purchasing.
- Page-level horizontal overflow, unbounded tables, color-only statuses or tooltip-only blockers.
- Generic `Đơn mua` copy where request/order distinction matters.
- Production local-path field, workbook upload surface or canonical refresh apply UI.
- Silent `KG` fallback, guessed supplier split, clamped dates or hidden ambiguous source data.
- New UI kit, font, icon family, decorative motion, gradients, glass effects or marketing cards.
