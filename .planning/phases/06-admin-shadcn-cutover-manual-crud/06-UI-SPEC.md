# Phase 6: UI Design Contract

> Visual and interaction contract for the `/admin-data` BOM workbench. Generated by gsd-ui-researcher and ready for gsd-ui-checker.

---

## Design Intent

This phase is a targeted hardening of a dense industrial administration surface. Preserve the current information architecture, IPC blue and slate light theme, compact controls, small radii and operational labels. The visual dials are fixed at variance 3 of 10, motion 2 of 10 and density 8 of 10.

The generic design search suggested marketing-oriented comparison-table and exaggerated-minimalism patterns. Those recommendations are rejected because they conflict with the locked operational admin context. No hero layout, oversized typography, decorative gradient, glass effect, new font or page redesign is permitted.

## Design System

| Property | Contract |
|----------|----------|
| Tool | Existing shadcn-style local primitives with the installed `shadcn` 4.11 package; no CLI initialization and no new preset |
| Preset | None; the repository has no `components.json` and this phase must not initialize or install a second system |
| Component library | Existing IPC primitives plus `@base-ui/react` 1.5 for accessible `Dialog` and `AlertDialog` foundations |
| Styling | Tailwind CSS 4 plus semantic IPC tokens from `frontend/src/styles/index.css` |
| Icon library | Lucide only, outline style, 14px to 16px in controls and 18px in section headings |
| Font | Existing `Inter`, `system-ui`, `Segoe UI`, `Roboto`, `Arial`, sans-serif stack |
| Radius | Existing 2px, 3px and 4px IPC radii only |
| Motion | Existing 120ms to 200ms focus, hover, pressed and state feedback only; honor reduced motion |

### Primitive contract

- Rebuild the shared dialog wrapper on Base UI parts already installed: `Root`, `Trigger`, `Portal`, `Backdrop`, `Viewport`, `Popup`, `Title`, `Description` and `Close`.
- Use Base UI `AlertDialog` for destructive apply and stop confirmations. Use regular `Dialog` for add and version forms.
- Do not add a modal, grid, table, icon, font, animation or state library.
- Use semantic IPC CSS variables. Do not introduce hardcoded per-component brand colors.
- Keep native table semantics. Do not add custom arrow-key grid behavior.

## Spacing Scale

All new Phase 6 spacing values use this permitted compact scale.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, action-button gap, status dot gap |
| sm | 8px | Inline controls, compact cell content, toolbar gaps |
| md | 16px | Table-cell horizontal padding, form field gaps, workbench grid gap and card padding |
| lg | 24px | Dialog section gap and major panel separation |
| xl | 32px | Dialog outer spacing on large screens only |
| 2xl | 48px | Reserved major section break; not used inside the dense table workbench |
| 3xl | 64px | Reserved page-level spacing; not used inside the dense table workbench |

Exceptions: existing 2px to 4px border radii and 32px to 36px control heights are dimensions, not spacing. Do not use the existing 18px or 28px spacing tokens for new Phase 6 layout.

## Typography

Use exactly four sizes and two weights inside the Phase 6 workbench.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Meta | 12px | 400 | 1.4 | Codes, source rows, helper text, secondary dates |
| Compact label | 13px | 600 | 1.35 | Table headers, form labels, buttons, status labels |
| Body | 14px | 400 | 1.4 | Table values, descriptions, validation and feedback |
| Dialog heading | 18px | 600 | 1.25 | Dialog title only |

Do not add a display style. Hashes and immutable identifiers may use the existing monospace stack at 12px or 13px without adding another weight.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant, about 60% | `#FFFFFF` / `--card` | Workbench, table, dialog and input surfaces |
| Secondary, about 30% | `#EEF2F6` / `--background` with `#F1F5F9` / `--ipc-slate-100` | Page background, setup rail, table header, disabled and skeleton surfaces |
| Accent, at most 10% | `#1A56A8` / `--ipc-primary` | Primary preview/apply CTA, selected tier or tab, focus-supported information links |
| Destructive | `#C53030` / `--ipc-danger` | Destructive CTA, blocker icon and destructive text only |
| Success | `#0F766E` / `--ipc-success` | Passed prerequisite, completed apply and published state with text label |
| Warning | `#C05621` / `--ipc-warning` | Stale preview, missing prerequisite and nonblocking warning with text label |
| Focus | `#2F6FED` / `--ipc-focus-ring` | Two-pixel visible focus outline with two-pixel offset |

Accent is reserved for the selected view, selected tier, primary preview/apply actions, focus-visible ring and intentional information links. It is not a blanket color for every interactive control. Status must always combine color with text and, where already used, an icon or dot.

## Information Architecture

### Preserved page shell

- Keep route `/admin-data` and the `BOM theo đơn giá` tab.
- Keep the current admin tab order and the two-column BOM workbench at the supported desktop width.
- Keep manual current-BOM and canonical-preview views in the right workbench.
- Do not alter unrelated Contract, Dữ liệu lỗi, Tồn kho, Thống kê, Audit or Nhân viên surfaces.

### Canonical bulk flow

The left setup rail has this fixed order:

1. Canonical workbook selector with accepted `.xlsx` guidance.
2. Customer scope selector, with `BOM global` first.
3. Effective-from date.
4. Required reconciliation reason when the Phase 5 contract requires it.
5. `Xem trước thay đổi` primary button.
6. Read-only helper text: `File phải có đủ ba sheet 25k, 30k và 34k. Xem trước không thay đổi dữ liệu.`

The three tier controls remain available as table/manual-CRUD view context. They do not redefine the canonical workbook contract and they must not narrow server manifest totals. The customer scope and effective date are server-validated preview parameters, not client authority.

The old `Commit` wording is not visible in the canonical workbench. The visible flow is `Xem trước thay đổi`, review the server manifest, then `Áp dụng BOM canonical` in the confirmation dialog. Old template/download helpers may remain only as clearly read-only compatibility helpers until Phase 7; they must not call an old writer or appear as the canonical import contract.

### Right workbench order

1. Dataset-wide scope summary.
2. View toolbar with `BOM hiện tại` and `Preview canonical`.
3. Current view tools: tier/scope display filters, search and `Thêm dòng`.
4. One fixed table viewport reused for current, preview, loading, error, empty and success states.
5. Contextual feedback region below the viewport only if it does not move the toolbar or viewport.

## Dataset-Wide Summary Contract

The summary is populated only from the Phase 4/5 server manifest. Search, tier display filter and rendered-row slicing cannot feed these values.

Show these labels in a compact wrapping strip or grid:

| Label | Source |
|-------|--------|
| `Tổng dòng` | Server manifest dataset total |
| `Không đổi` | Canonical `Unchanged` count |
| `Tạo mới` | Canonical `Create` count |
| `Tạo version` | Canonical `Version` count |
| `Lưu trữ / Ngừng` | Legacy `archive` plus `deactivate`, kept separately in accessible detail |
| `Xóa orphan` | Legacy `delete` count |
| `Tái tạo draft` | Downstream `regenerate` count |
| `Giữ lịch sử` | History-kept count |
| `Blocker` | Server blocker count |

Do not merge the three action vocabularies in state or copy: canonical actions are `Unchanged`, `Create`, `Version`; legacy actions are `keep`, `archive`, `deactivate`, `delete`, `block`; downstream action is only `regenerate`.

The summary keeps the same block size while values load. Skeletons use fixed-width placeholders. Searching for a dish changes only visible table rows and the optional text `Hiển thị N dòng`, never the summary values or apply payload.

## Layout Contract

### Workbench grid

- At viewport widths 1280px and above: `grid-template-columns: minmax(280px, 360px) minmax(0, 1fr)` with a 16px gap.
- Below 1280px: stack setup rail above the right workbench. Do not compress the table columns to fit the page.
- Both grid children use `min-width: 0` so the right workbench cannot enlarge the page shell.
- The page shell must not gain horizontal overflow at 1365x900, 1280x900 or 390x844.

### Toolbar

- Desktop toolbar minimum block size is 44px and its controls stay vertically centered.
- Mobile toolbar stacks controls in document order with an 8px gap.
- View tabs remain first, search second and `Thêm dòng` last.
- Toolbar position and outer dimensions are unchanged across full, searched, loading, error and empty states.

### Fixed table viewport

- Use one `BomTableViewport` or hardened `DataTableShell` with `width: 100%`, `height: 520px`, `min-height: 520px`, `max-height: 520px`, `overflow: auto`, `overscroll-behavior: contain` and `scrollbar-gutter: stable` at desktop targets.
- On narrow screens, use a documented fixed 440px block size. The same size applies to every data state at that viewport.
- Keep the table header sticky at `top: 0` with the existing opaque slate header surface and z-index 5.
- The viewport is a focusable named region. Visible focus uses the IPC ring.
- Loading, error, empty and no-search-result presentations render inside the same table and `tbody`; do not replace the shell with a card of different size.
- Horizontal and vertical scrollbars belong to the viewport. The page shell never scrolls horizontally because of the table.

### Stable state geometry

- Header height and all column widths have a maximum one-pixel difference across full, searched, loading, error and empty fixtures at the same viewport.
- Loading shows the real header plus at least six reserved skeleton rows using the normal row height.
- Empty and error states use one full-width body row but retain the `colgroup`, header and 520px or 440px viewport.
- Search no-result is distinct from dataset empty and never changes server totals.
- Do not use content-driven column width, auto table layout or conditional outer wrappers.

## Search Contract

- The visible label is available to assistive technology: `Tìm món hoặc nguyên liệu`.
- Placeholder: `Tìm món, nguyên liệu...`.
- Desktop width: minimum 240px, preferred 280px, maximum 320px. It may become full-width in the stacked toolbar.
- Input height: 36px.
- Lucide `Search` icon: 16px, `aria-hidden="true"`, `pointer-events: none`, absolute left inset 8px, vertical center at 50% with translateY minus 50%.
- Input left padding: 36px minimum. The icon's right edge and the first text pixel retain at least 8px clearance.
- Search filters only the already-renderable display collection. It does not modify manifest ID, source hash, preview counts, candidate IDs, apply scope or request payload.
- While filtering, the toolbar, table viewport, `colgroup` and scroll ownership remain unchanged.
- If no row matches, show `Không có dòng phù hợp.` and `Xóa từ khóa hoặc đổi bộ lọc. Phạm vi áp dụng không thay đổi.` inside the stable table body.

## Row Action Contract

- Action cell width is fixed at 168px on the current table.
- The inner group is `inline-flex`, `flex-flow: row nowrap`, `justify-content: center`, `gap: 4px` and `white-space: nowrap`.
- Both buttons use `flex: none`, `white-space: nowrap`, `overflow-wrap: normal` and never shrink.
- `Sửa` minimum width is 68px. `Ngừng` minimum width is 76px. Icons are 14px and never wrap above labels.
- At 1365px and 1280px viewports, both actions have the same top coordinate and remain in one row. At smaller widths the table scrolls locally instead of wrapping actions.
- Use a Pencil icon for `Sửa` and Power icon for `Ngừng`; no emoji and no alternate icon family.
- A row refreshed after version creation retains a stable logical row anchor so focus can return to the new version's `Sửa` trigger. If stopping removes the row from the active view, return focus to the table region and announce the result.

## Manual CRUD Contract

### Add line

- Trigger: `Thêm dòng`.
- Title: `Thêm dòng BOM thủ công`.
- Preserve fields for dish, ingredient, quantity per serving, waste percentage, status, effective-from, effective-to and reason/note.
- Scope and tier are visible in the dialog description and come from the current workbench context.
- Primary CTA: `Thêm dòng BOM`.
- Successful creation returns to current BOM view, focuses the created row or its edit trigger and announces `Đã thêm dòng BOM thủ công.`

### Create version

- Trigger: `Sửa`.
- Title: `Tạo version mới cho dòng BOM`.
- Published dish identity stays immutable. Edited technical values create a new version and never overwrite published history.
- Reason is required, trimmed, nonblank and 1 to 500 characters. The server derives actor identity; the UI never accepts an actor field from the operator.
- Primary CTA: `Lưu version mới`.
- Description includes: `Dòng đã published được điều chỉnh bằng version mới để giữ lịch sử.`
- Success copy: `Đã tạo version mới cho dòng BOM. Lịch sử cũ vẫn được giữ nguyên.`

### Stop line

- Use Base UI `AlertDialog`.
- Trigger: `Ngừng`.
- Title: `Ngừng áp dụng dòng BOM?`.
- Description: `Dòng BOM sẽ ngừng hiệu lực nhưng không bị xóa. Lịch sử và audit vẫn được giữ nguyên.`
- Show dish, ingredient, scope, tier and current effective interval.
- Require a trimmed nonblank reason of 1 to 500 characters when the Phase 5 published-line contract applies.
- Cancel CTA: `Hủy`. Destructive CTA: `Ngừng áp dụng`.
- Initial focus is `Hủy`. The destructive CTA uses the danger token, not the blue primary style.

### Form behavior

- Field errors appear adjacent to their field and are linked with `aria-describedby`.
- The first invalid field receives focus after submission.
- Save buttons show a text loading label and remain disabled while the request is active.
- Form dialogs cannot close through the backdrop while saving. Escape is safe before submission and disabled while a mutation cannot be cancelled safely.

### Gating matrix

The destructive CTA is enabled only when every condition is true:

- Manifest is present and belongs to the selected file, customer scope and effective date.
- Preview is fresh according to the server.
- Blocker count equals zero.
- Backup marker is present and verified.
- Gate C evidence is present and passed.
- Apply feature is enabled by the Phase 5 server configuration.
- No preview or apply request is running.

Show all failing reasons as visible text above the CTA. Exact messages:

- `Preview đã cũ. Hãy chạy lại preview trước khi áp dụng.`
- `Còn {N} blocker cần xử lý.`
- `Chưa có backup hợp lệ.`
- `Chưa có bằng chứng Gate C đạt yêu cầu.`
- `Chức năng áp dụng đang bị khóa bởi cấu hình an toàn.`

The client submits the manifest token and operator reason defined by Phase 5. It never submits filtered candidate IDs, client-computed action counts, an actor identity or a filesystem path.

### Focus and dismissal

- Initial focus is the `Hủy` button.
- Focus is trapped while open.
- Escape and backdrop dismissal are safe before apply begins.
- Once apply begins, pointer dismissal, Escape and close controls are disabled. The progress state cannot be dismissed accidentally.
- On cancel, return focus to the exact `Áp dụng BOM canonical` trigger.
- On success, close only after the success response is captured, return focus to the apply trigger or stable summary anchor, refresh current server data and announce the result.

## State and Feedback Contract

| State | Visual and text contract | Interaction contract |
|-------|--------------------------|----------------------|
| No preview | `Chưa có preview.` plus `Chọn workbook rồi bấm Xem trước thay đổi.` | Apply trigger absent or disabled with reason |
| Preview loading | `Đang tạo preview...` with fixed skeleton rows | Inputs remain visible; repeated preview disabled |
| Preview success | Dataset summary, table rows and `Preview còn hiệu lực` | Apply eligibility follows server prerequisites |
| Preview stale | Warning icon plus stale copy | Apply disabled; preview rerun is primary recovery |
| Preview error | `Không tải được preview.` plus `Kiểm tra file và kết nối, sau đó thử lại. Dữ liệu chưa thay đổi.` | Contextual `Thử lại` button keeps file/scope/date selections |
| Dataset empty | `Chưa có dòng BOM trong phạm vi này.` plus `Chọn phạm vi khác hoặc thêm một dòng BOM.` | `Thêm dòng` stays available |
| Search empty | `Không có dòng phù hợp.` plus scope-preservation copy | Clear search or change display filter |
| Apply loading | `Đang áp dụng...` and live progress text | Dialog locked against accidental dismissal |
| Apply success | Success icon and exact success copy | Refresh current catalog/report and preserve focus |
| Mutation error | `Chưa thể lưu thay đổi.` plus server-safe reason and `Thử lại` | Dialog remains open with entered reason preserved |

All feedback uses visible text and an existing semantic token. `role="alert"` is reserved for errors or blockers needing immediate attention; nonurgent status uses `role="status"` or `aria-live="polite"`. Do not communicate success, warning or failure by color alone.

## Accessibility and Keyboard Contract

- Each dialog uses programmatically associated Base UI Title and Description parts. `aria-label` alone is not sufficient.
- Normal tab order follows: view tabs, setup fields, preview action, summary, current/preview tabs, search, add, table region, row actions.
- The table viewport remains a native focusable region with a descriptive accessible name. No custom cell-grid keyboard model is added.
- Every icon paired with visible text is `aria-hidden`; icon-only controls require an accessible name.
- Focus remains visible on every interactive element and is never clipped by the table viewport or dialog.
- Dialog focus returns to the logical triggering control after cancel, save, stop or apply.
- Row update announcements include the dish and ingredient when available.
- Error summary links or focuses the first invalid field.
- Disabled destructive actions expose visible prerequisite reasons. Do not rely on a tooltip that keyboard or touch users may miss.
- Preserve 4.5 to 1 contrast for normal text and 3 to 1 for large text, component borders and state indicators.
- Reduced motion removes nonessential transitions. No animation may be required to understand a state change.

## Phase 5 Safety Prerequisites

Phase 6 may be implemented only against executed Phase 5 contracts. The UI must not invent endpoint DTOs or infer safety from the rendered table.

Required upstream evidence:

- Gate A and Gate B passed.
- Gate C passed on an identified isolated clone with a verified backup and restore-on-failure proof.
- Canonical preview manifest includes source hash, policy and contract versions, effective date, DB fingerprint, TTL/freshness, server counts, blocker reasons, backup marker and Gate C evidence.
- Apply remains disabled by server configuration until Gate C is passed.
- Apply reparses and reclassifies server-side in one transaction and treats UI values as informative only.
- Manual published edit and stop endpoints enforce server-derived actor, bounded nonblank reason, versioning and atomic audit.
- Cache/report invalidation and downstream draft regeneration are available from Phase 5 before the UI advertises success.

If any prerequisite is absent, the Phase 6 surface can show read-only current BOM data and manual actions that are already safe, but it must show canonical apply as unavailable with the server-provided reason.

## Verification Contract

### Unit and component tests

- Search changes rendered rows only. Manifest totals, manifest token and apply payload remain byte-for-byte equivalent before and after filtering.
- Changing file, customer scope or effective date invalidates preview immediately and disables apply.
- Apply payload contains no client candidate IDs, actor identity, action counts or local path.
- Add, version and stop forms preserve their fields, reason rules and server error copy.
- Dialogs expose accessible names and descriptions through `aria-labelledby` and `aria-describedby`.
- Apply and stop alert dialogs focus `Hủy` first and return focus correctly.
- Every false safety prerequisite disables apply and renders the matching reason.

### Playwright geometry tests

Run targeted `/admin-data?view=bom-import` fixtures at 1365x900, 1280x900 and 390x844.

- Capture the toolbar and table viewport bounding boxes in full, searched, loading, error and empty states. At the same viewport, x, y, width and height differ by no more than one pixel.
- Capture every table header width across the same states. Each width differs by no more than one pixel.
- Assert the desktop table viewport height is 520px and narrow viewport height is 440px.
- Assert table `scrollWidth` may exceed `clientWidth`, but document `scrollWidth` never exceeds viewport width by more than two pixels.
- Assert the table owns vertical scrolling when rows exceed the viewport.
- Assert search input left padding is at least 36px and the first text area begins at least 8px after the icon's right edge.
- Assert `Sửa` and `Ngừng` share the same row, have equal top coordinates within one pixel and neither button has wrapped or clipped text at 1365px and 1280px.
- Assert full source hash and all safety counts are visible in the apply dialog.

### Keyboard and safety tests

- Open each add, version, stop and apply dialog with the keyboard, cancel with Escape when safe and verify focus returns.
- While apply is running, Escape, backdrop and close controls do not dismiss the dialog.
- A blocker, stale preview, missing backup, missing Gate C evidence or disabled server feature individually disables apply.
- Search for one dish, then clear search. Server totals, scope and apply eligibility do not change.
- Complete add, version and stop flows in global and customer scope fixtures and verify published history remains visible through the server contract.

### Existing quality gates

- `npm run lint --workspace frontend`
- `npm run build --workspace frontend`
- `npm run test:unit --workspace frontend`
- `npm run test:smoke --workspace frontend`
- `npm run test:ui-audit --workspace frontend`

## Forbidden Patterns

- Client totals derived from filtered `currentBomRows` or a preview slice.
- Candidate IDs or destructive counts submitted from the client as authority.
- Auto table layout or content-driven columns.
- Page-level horizontal overflow caused by the BOM table.
- A loading, error or empty card that replaces the fixed table shell.
- Wrapped or shrinking `Sửa` and `Ngừng` actions.
- A search icon with no reserved input padding.
- Dialogs built only with `createPortal` and `aria-label` without focus trap, title, description and focus restoration.
- Blue styling for destructive confirmations.
- Backdrop or Escape dismissal during a noncancellable apply.
- Silent unknown state, color-only feedback, decorative motion, gradients, glass effects or new typography.
- Any Phase 7 deletion of legacy endpoints, DTOs or backend consumers in this phase.
