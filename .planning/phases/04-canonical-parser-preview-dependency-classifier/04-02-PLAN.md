# Phase 4 Plan 02: Pure lifecycle classifier and complete dependency evidence

<objective>
Create two deliberately separate immutable decision models: canonical desired-vs-current reconciliation with `Unchanged`, `Create`, and `Version`, and legacy lifecycle classification with `keep`, `archive`, `deactivate`, `regenerate`, `delete`, and `block`. Feed the legacy model complete read-only dependency evidence; neither model may perform or invoke mutation.
</objective>

<execution_context>
- Phase 3 final Gate A and Plan 04-01 parser/security tests must pass first. Use the actually delivered `03-LIFECYCLE-POLICY.md` and Phase 3 policy version; do not invent a parallel status map.
- Read `04-CONTEXT.md` decisions D-04-04 and D-04-07, `04-PATTERNS.md` sections 5 and 7, the Phase 3 lifecycle policy, `WorkflowReportService` read-side dependency queries and mixed fixtures before editing.
- Run upstream GitNexus impact immediately before each existing context/service/test symbol edit. HIGH/CRITICAL requires user approval; UNKNOWN is a lower-bound requiring direct consumer review.
- Before every commit, compare immutable Phase 3 baseline with `.artifacts/bom-v1.1/gate-b/phase4-preedit/owned-paths.txt` and run `node .gitnexus/run.cjs detect_changes -r IPCManagement --scope all`. Do not stage unrelated dirty files.
- This plan contains no parser changes, endpoint, manifest/apply path, `SaveChanges`, audit insertion, cleanup DML, frontend/RTK/UI or old-surface removal.
</execution_context>

<must_haves>
- D-04-04: every dependency read uses `AsNoTracking`, scalar/immutable projections and no tracked mutation or `SaveChanges`.
- D-04-07: preview and future apply share one pure policy/action vocabulary; unknown lifecycle/reference/stock/evidence completeness always maps to `block`.
- Canonical diff uses the exact key `(dish, ingredient, unit, tier, customer, effective interval)` and only `Unchanged`, `Create`, `Version`; it is never conflated with the legacy cleanup vocabulary.
- Customer override matching is ingredient-level: matching customer rows overlay global while unmatched ingredients fall back to global.
- Classifier accepts no `DbContext`, EF entity, lazy navigation or service that can mutate.
- `delete` requires affirmative true-orphan proof, zero stock and zero retained/downstream reference; absence from workbook/name/code is never sufficient.
- `archive`/`deactivate` conservatively preserve catalog/history, while `regenerate` is only a Phase 4 label and performs no downstream work.
</must_haves>
